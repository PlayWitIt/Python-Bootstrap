What is PyBootstrap?

PyBootstrap is a tool that eliminates the tedious setup work when starting a new Python project. Instead of manually creating virtual environments, installing dependencies, and configuring Git - you run one command and start coding.

🎯 The Goal: Go from "I want to write Python" to "running code" in seconds, not minutes.

The Problem It Solves

Every Python project requires the same setup steps:

  1. Create project directory
  2. python -m venv .venv - Create virtual environment
  3. source .venv/bin/activate - Activate it
  4. Install dependencies with pip
  5. Create requirements.txt
  6. Create .gitignore
  7. Initialize Git repository

That's 7 steps before writing your first line of code!

The PyBootstrap Solution

pybs init
→
./py_bootstrap.sh
→
./Run.sh
→
💻 Code!

That's it. Three commands max, and after the first time, just ./Run.sh.

How It Works: Step by Step

Step 1: Initialize (One Time)

# Install pybootstrap
pip install pybootstrap

# Initialize with both scripts (default)
pybs init --dir my_project
pybs init                    # Current directory
pybs init --force            # Overwrite existing

# Or create only one script:
pybs bootstrap --dir my_project   # Just py_bootstrap.sh
pybs run --dir my_project           # Just Run.sh

# Or run scripts directly (no Run.sh needed):
pybs exec myscript.py
pybs exec myscript.py --arg value  # Pass arguments
pybs exec --no-venv                # Use system Python
# Download py_bootstrap.sh and Run.sh from this repo
# Or copy them from an existing project

Step 2: Setup Environment (One Time Per Project)

./py_bootstrap.sh

This script will:

  • Detect your Python (pyenv, conda, or system)
  • Create a virtual environment (.venv)
  • Install dependencies from requirements.txt
  • ✨ Auto-detect dependencies if requirements.txt is missing (using pipreqs)
  • Offer to initialize Git with .gitignore

🤖 Magic Feature: Auto-Detect Dependencies

No requirements.txt? No problem!

PyBootstrap uses pipreqs to automatically scan your Python code and generate a requirements.txt file. It reads your imports and finds the exact packages you need. Just say "yes" when asked!

# What happens when requirements.txt is missing:
# 1. PyBootstrap asks: "Auto-detect dependencies?"
# 2. You say: "Yes"
# 3. pipreqs scans your code
# 4. requirements.txt is created automatically
# 5. Dependencies are installed
# Done!

Step 3: Run Your Code (Every Time)

./Run.sh

Run.sh automatically:

  • Finds your virtual environment
  • Activates it
  • Locates your Python script (main.py, app.py, etc.) in any subdirectory
  • Shows interactive menu if no default script found
  • Can save your selection as the default for next time
No more remembering: Which Python? Which venv? What was the script name? Just run ./Run.sh

Understanding the Components

📦
pybs (CLI)

The command-line tool that initializes projects and builds the executable. Built with Click and packaged with PyInstaller.

⚙️
py_bootstrap.sh

The setup script. Run once per project. Creates venv, installs deps, sets up Git.

▶️
Run.sh

The runner script. Run every time you want to execute your code. Finds venv and script automatically.

📋
templates/

Folder bundled into pybs executable. Contains py_bootstrap.sh and Run.sh to copy into new projects.

✨
pipreqs (Under the Hood)

The secret sauce! When you don't have a requirements.txt, PyBootstrap uses pipreqs to automatically scan your Python code, detect what you import, and generate the exact dependencies you need.

No more manually listing packages!

File Structure

your-project/
├── .venv/                  # Virtual environment (created by py_bootstrap.sh)
├── templates/              # Bundled templates (not needed after init)
├── main.py                # Your code
├── requirements.txt       # Your dependencies
├── py_bootstrap.sh        # Setup script (from pybs init)
├── Run.sh                 # Runner script (from pybs init)
└── .git/                  # Git repository (optional)

How They Work Together

Technical Deep Dive

How pybs init Works

When you run pybs init, here's what happens:

  1. Find templates: Locates the bundled templates/ folder (works both from source and PyInstaller executable)
  2. Copy files: Copies py_bootstrap.sh and Run.sh (or just one based on command)
  3. Make executable: Sets execute permissions on the copied scripts
  4. Print instructions: Shows the user next steps
Available Commands:
  • pybs init - Creates both scripts (default)
  • pybs bootstrap - Creates only py_bootstrap.sh
  • pybs run - Creates only Run.sh
  • pybs exec script.py - Run scripts directly

How py_bootstrap.sh Works

Python Detection Order

  1. Check for .python-version file → use pyenv
  2. Check for conda installation → use conda
  3. Fallback to system Python
# The detection logic in py_bootstrap.sh:
if command -v pyenv &> /dev/null && [ -f .python-version ]; then
    # Use pyenv
elif command -v conda &> /dev/null; then
    # Use conda
else
    # Use system Python
fi

Virtual Environment Creation

Uses Python's built-in venv module:

python -m venv .venv

Then activates it:

source .venv/bin/activate

Dependency Installation

  1. If requirements.txt exists → install from it
  2. If requirements.txt is empty/missing → offer to use pipreqs to auto-detect

🤖 How pipreqs Works

pipreqs scans your Python files and:

  1. Parses all import and from ... import statements
  2. Maps them to their PyPI package names
  3. Generates a clean requirements.txt
# Example pipreqs command:
pipreqs . --force --ignore .venv,tests,docs

# The --ignore flag skips folders you don't want scanned

PyBootstrap runs this automatically when you say "yes" - you don't even see the command!

Git Initialization

Offers to:

  • Initialize a new repository
  • Create .gitignore with Python defaults
  • Set default branch to "main"

How Run.sh Works

# Simplified logic:
1. Find venv (.venv or venv)
2. Find script: main.py → app.py → myapp.py (in any subdirectory)
3. If no default found: show interactive menu of all .py files
4. Activate venv
5. Run script with any args passed to Run.sh
6. Ask to save selection as default for next time

✨ New: Color-Coded Output

Run.sh now features color-coded output for easy reading:

  • [INFO] - General information
  • [SUCCESS] - Successful operations
  • [ERROR] - Errors and failures
  • === Section === - Section headers

Building the Executable

PyInstaller bundles everything into a single executable:

pyinstaller --onefile --add-data "templates:templates" --name "pybs" main.py
Flag Purpose
--onefile Bundle everything into a single executable
--add-data Include templates folder in the bundle
--name Name of the output executable

Frequently Asked Questions

Q: Is this safe? What if it breaks my Python?

A: Completely safe! PyBootstrap uses virtual environments, which are isolated from your system Python. Nothing you do with these scripts affects other Python projects.

Q: Can I use this on Windows?

A: The scripts are bash (Linux/macOS), but work great on Windows via WSL (Windows Subsystem for Linux). For native Windows, you can use Git Bash or the scripts in WSL.

Q: What if I don't have requirements.txt?

A: No problem! PyBootstrap uses pipreqs to automatically scan your code and generate requirements.txt for you. Just say "Yes" when asked - it does all the work!

Q: Can my script be in a subfolder?

A: Yes! Run.sh scans all subdirectories to find your Python scripts. If it can't find main.py/app.py/myapp.py, it shows an interactive menu with all available .py files. You can also type a custom path to a script in any subfolder.

Q: Can I customize the scripts?

A: Absolutely! The scripts are yours. Modify them to fit your workflow. You can even commit your customized versions to share with your team.

Q: Do I need to run py_bootstrap.sh every time?

A: No! Just once per project. After that, just use ./Run.sh to execute your code.

Q: What if my venv gets corrupted?

A: Just delete the .venv folder and run ./py_bootstrap.sh again. It's that simple.

Q: Can I use this with Docker?

A: Definitely! Use PyBootstrap to set up locally, then commit your .venv or convert to a Dockerfile. Or use the scripts inside Docker.
